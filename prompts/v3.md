# mee6 v3 — Rails Migration Plan

## Vision

Replace the Python/FastAPI monolith with a Ruby on Rails application. Rails owns
all orchestration, scheduling, persistence, and the React frontend. Two external
services handle what Rails cannot do natively: a Python agent service for LLM,
Browser, and Calendar agents; and a standalone WhatsApp service (Python + neonize)
that is fully decoupled and communicates with Rails via a clean REST contract.
Each service has a single, replaceable responsibility.

## Design Patterns

| Pattern | Where used |
|---------|-----------|
| **Service Object** | `Pipelines::ExecutorService`, `Triggers::DispatchService` — business logic isolated from controllers |
| **Repository via ActiveRecord** | Models are thin data holders; queries live in scopes or service objects, never in controllers |
| **Strategy (via HTTP)** | Each agent type is a strategy; Rails dispatches by `agent_type` string to the service, which resolves the implementation |
| **Observer / Webhook** | Python service POSTs WhatsApp events to Rails; Rails enqueues `PipelineJob` with the message as initial input |
| **Background Job** | `PipelineJob` (Sidekiq) runs pipelines asynchronously; `CronJob` fires on schedule |
| **Presenter / Serializer** | JSON responses built with dedicated serializer classes, not `as_json` in controllers |
| **Feature module namespacing** | `Pipelines::`, `Triggers::`, `Integrations::` — keep domain boundaries explicit |

## Database Schema (Rails models)

Migrate directly from the existing PostgreSQL schema. No data loss required
for pipelines, triggers, run records, memory, calendars, and WhatsApp groups.
`whatsapp_messages` table is dropped (inbound storage removed per event-driven
refactor; outbound audit trail moved to `run_records` summary if needed).

| Model | Rails table | Notes |
|-------|------------|-------|
| `Pipeline` | `pipelines` | `has_many :steps` |
| `PipelineStep` | `pipeline_steps` | `belongs_to :pipeline`, ordered by `step_index` |
| `Trigger` | `triggers` | polymorphic via `trigger_type` enum: `cron`, `whatsapp`, `wa_group` |
| `RunRecord` | `run_records` | append-only execution history |
| `Memory` | `memory` | config row per named memory bucket |
| `MemoryEntry` | `memory_entry` | `belongs_to :memory`, TTL enforced on read |
| `Calendar` | `calendars` | Google Calendar config |
| `WhatsAppGroup` | `whatsapp_groups` | synced from agent service |
| `WhatsAppSetting` | `whatsapp_settings` | single-row config (phone number) |

## Agent Service HTTP Contract

All agent execution goes through a single endpoint. Rails does not know how
any agent works internally.

### Run an agent step

```
POST /agents/run
Authorization: Bearer <AGENT_SERVICE_SECRET>

{
  "agent_type": "llm_agent",
  "config": { "prompt": "Summarise: {input}", "model": "gpt-4o" },
  "input": "the previous step output"
}

→ 200 { "output": "..." }
→ 422 { "error": "..." }
→ 504 (timeout)
```

### Agent metadata (for pipeline editor)

```
GET /agents/schema
→ 200 {
  "llm_agent":    { "label": "LLM Agent",  "fields": [...] },
  "whatsapp_send":{ "label": "WhatsApp Send", "fields": [...] },
  ...
}
```

## WhatsApp Service Contract

The WhatsApp service is a standalone Python + neonize process. Rails is its only
client. It has no knowledge of pipelines or triggers — it only knows about
phones, groups, and messages.

### Session management

```
GET  /status
→ { "status": "connected|pending_qr|disconnected", "qr_svg": "..." }

POST /connect
POST /disconnect
GET  /groups
→ [{ "jid": "120363xxx@g.us", "name": "Family" }, ...]
```

### Registration (Rails → WhatsApp service, called on Rails boot)

Rails tells the service which phones and groups to monitor, and provides its
own callback URL. The service stores this and calls the callback whenever a
matching message arrives.

```
POST /monitor
Authorization: Bearer <shared secret>

{
  "callback_url": "https://rails-app/webhooks/whatsapp",
  "phones":     ["34612345678", "34699000000"],
  "group_jids": ["120363xxx@g.us"]
}

→ 200 { "ok": true }
```

Rails re-registers on every boot so the two services stay in sync after a
restart of either one. The WhatsApp service replaces any previous registration.

### Inbound message (WhatsApp service → Rails)

```
POST <callback_url>
X-Webhook-Secret: <shared secret>

{ "type": "dm",    "sender": "34612345678",       "text": "hello" }
{ "type": "group", "chat_jid": "120363xxx@g.us",  "text": "hello" }
```

Rails finds all enabled matching triggers and enqueues `PipelineJob` for each
with `initial_input: text`. No message is stored. If the user wants persistence,
they add a Memory Agent step to the pipeline.

### Send a message (Rails → WhatsApp service)

```
POST /send
Authorization: Bearer <shared secret>

{ "type": "dm",    "phone": "+34612345678",      "message": "..." }
{ "type": "group", "group_jid": "120363xxx@g.us", "message": "..." }

→ 200 { "ok": true }
→ 422 { "error": "..." }
```

The WhatsApp Send and WhatsApp Group Send pipeline steps call this endpoint
through `Integrations::WhatsAppClient` in Rails. The agent service is not
involved.

## Agent Ownership

| Agent | Runs in | Reason |
|-------|---------|--------|
| LLM Agent | Python agent service | Python LLM SDK |
| Browser Agent | Python agent service | Playwright (Python) |
| Calendar Agent | Python agent service | Anthropic tool-use loop + Google Calendar Python SDK |
| **WhatsApp Send** | **Rails → WhatsApp service** | Calls `POST /send` on the WhatsApp service |
| **WhatsApp Group Send** | **Rails → WhatsApp service** | Calls `POST /send` on the WhatsApp service |
| **Memory Agent** | **Rails (native)** | Pure ActiveRecord — no external call |
| **Debug Agent** | **Rails (native)** | Rails logger — no external call |

Rails executes Memory and Debug natively. WhatsApp Send/Group Send call the
dedicated WhatsApp service. Everything else goes through the Python agent service.

`Pipelines::ExecutorService` resolves which path to take per `agent_type` before
executing each step — no routing logic leaks into the job or controller layer.

## Scheduling

Cron triggers are managed by `sidekiq-cron`. On app boot (and after any trigger
create/update/delete), `Triggers::SchedulerService` syncs the DB state to the
Sidekiq-cron schedule. Each cron entry calls `CronDispatcherJob` which enqueues
`PipelineJob`.

WhatsApp and WA group triggers are not scheduled — they fire reactively from
the webhook.

`run_now` (manual run) enqueues `PipelineJob` directly.

## Frontend (React)

Single-page application served from Rails' `public/` directory, built by Vite.
Communicates exclusively with `/api/v1/`. No server-rendered HTML except the
root `index.html`.

```
frontend/
  src/
    api/           # fetch wrappers per resource (pipelines, triggers, etc.)
    components/
      ui/          # Button, Card, Table, Badge — design system primitives
      pipeline/    # PipelineEditor, StepCard, FieldRenderer, AgentSelector
      triggers/    # TriggerList, TriggerForm (cron, whatsapp, wa_group)
      integrations/ # CalendarsSection, MemoriesSection, WhatsAppSection
      layout/      # AppShell, Sidebar, TopBar
    pages/
      DashboardPage.tsx
      PipelinesPage.tsx
      PipelineEditPage.tsx
      TriggersPage.tsx
      RunHistoryPage.tsx
      IntegrationsPage.tsx
    hooks/
      usePipelines.ts
      useTriggers.ts
      useRunRecords.ts
    router.tsx
    main.tsx
```

Use **React Query** for all server state (cache, refetch, mutations). Avoid
Redux. Form state is local `useState`; complex forms use `react-hook-form`.

The pipeline editor fetches the agent schema from `/api/v1/agents/schema` on
mount and renders fields dynamically — same data-driven approach as v2 but
entirely in React.

## Migration Phases

### Phase 1 — Rails foundation
- New Rails app in `/rails` directory (API mode + Vite for React)
- PostgreSQL connection; mirror existing schema via migrations
- Devise or simple token auth if needed
- Sidekiq + Redis configured
- Health check endpoint

### Phase 2 — Agent service contract
- Define and document the HTTP contract (schema above)
- Wrap existing Python plugins behind a thin FastAPI router (`/agents/run`, `/agents/schema`, `/agents/whatsapp/*`)
- `AgentClient` service in Rails (Faraday or Net::HTTP, with timeout + error handling)
- Integration test: Rails calls agent service, agent service responds

### Phase 3 — Pipeline execution engine
- `Pipeline`, `PipelineStep` models and CRUD API
- `Pipelines::ExecutorService`: iterate steps, call `AgentClient#run` (or `Memories::AgentService` for memory steps), chain output
- `PipelineJob`: wraps executor, writes `RunRecord` on completion/error
- `run_now` endpoint

### Phase 4 — Scheduling
- `Trigger` model with `trigger_type` enum
- `Triggers::SchedulerService`: sync cron triggers to sidekiq-cron on boot and after mutations
- `CronDispatcherJob` → `PipelineJob`
- CRUD API for triggers

### Phase 5 — WhatsApp integration
- `Integrations::WhatsAppClient` service (HTTP client for the WhatsApp service)
- Registration on Rails boot: `POST /monitor` with current phones/groups + callback URL
- Webhook endpoint (`POST /webhooks/whatsapp`) — validates secret, calls `Triggers::DispatchService`
- `Triggers::DispatchService`: find matching enabled triggers by sender/jid, enqueue `PipelineJob` with `initial_input`
- WhatsApp Send / Group Send pipeline steps call `WhatsAppClient#send`
- WhatsApp management UI proxied through Rails (`/api/v1/integrations/whatsapp/status`, `/connect`, `/groups`)

### Phase 6 — Remaining integrations
- Memory: `Memory`, `MemoryEntry` models; `Memories::AgentService` (read, store, list); CRUD API
- Calendar: proxy config management to Rails; agent service handles the actual Google Calendar calls
- Integrations page API endpoints

### Phase 7 — React frontend
- Vite + React + TypeScript setup
- Design system primitives (match existing visual style)
- Pages: Dashboard, Pipelines, Pipeline editor, Triggers, Run history, Integrations
- Pipeline editor: fetch schema from agent service via Rails, render fields dynamically
- Replace all Jinja templates

### Phase 8 — Cutover
- Data migration script: copy rows from v2 PostgreSQL to v3 (pipelines, steps, triggers, run_records, memory, calendars, wa_groups)
- `whatsapp_messages` table is not migrated (dropped per event-driven refactor)
- Side-by-side run in staging; verify parity
- DNS/proxy cutover; decommission v2 FastAPI app

## Key Decisions to Carry Forward

- **WhatsApp inbound messages are never stored automatically.** The Python service fires a webhook; Rails dispatches pipelines with message text as `{input}`. Storage is opt-in via the Memory Agent step.
- **WhatsApp is a fully decoupled microservice.** It knows nothing about pipelines or triggers. Rails registers what to monitor and provides a callback; the service calls back on arrival and exposes a send endpoint.
- **Rails re-registers with the WhatsApp service on every boot.** This keeps them in sync after restarts without any persistent state on the WhatsApp side.
- **WhatsApp session state lives inside the WhatsApp service.** Rails only knows status and QR code from `GET /status`.
- **Memory and Debug agents are Rails-native.** They never make an external HTTP call.
- **WhatsApp Send/Group Send call the WhatsApp service directly**, not the Python agent service.
- **No server-rendered views.** Rails is API-only; React owns all UI.
- **Schema is the source of truth for the pipeline editor.** `GET /agents/schema` drives field rendering; no hardcoded field lists in the frontend.
